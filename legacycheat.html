This is Git Revision

What I'd normally use

  1. git init: to make a git repository
    • REMINDER: from here ownwards, it is paired with GitHub
  2. git add . : add all files
  3. git commit -m "-insert message here-" : short description of what we have done
  4. git push: push to remote repo in GitHub

Checking Git History

  1. git log: view all commits
  2. git chekout -commit hash(the weird number and alphabets thingy)- : goes back to that point of commit

Branching

Must add and commit in order to have the main(master) branch show up under git log

Recall: when we git init, that is considered as our main branch. Now, we are going to make different versions (branches) of the main branch

  1. git branch: to show what branch you are working on
  2. git checkout -insert branch name here-: work on this branch
    • git checkout main: to check back to main branch
  3. git branch -insert branch name here- : to switch to that branch
  4. git merge -insert branch name here-: merge branch here into main branch
  5. git branch -d -insert branch name here-: delete branch

An example of making a different branch

    • Rule of thumb is to branch on significant COMPONENT changes
  1. git branch bluecss: making a new branch
  2. git checkout bluecss: using this branch
  3. make changes within the files
  4. git add . and git commit -m "..." to save changes on the timeline
    • Note that the changes made on affects bluecss branch timeline and not the main(master) timeline
  5. git merge bluecss: merge into main branch if we liked the changes (Danger: overwrites the master branch)

Understanding git merge and conflicts

  1. If there is a merge conflict, it would show the error and comments on merge
  2. Make changes and once it is done to merge
  3. git add . and git commit which would give you a weird page
  4. :wq(save and close the file) on that page to automaticaaly merge the page

Git Collaboration

To note: In Git, "origin" is a shorthand name for the remote repository that a project was originally cloned from.

  1. git pull origin master: update the master branch with updated commits from collaborators
  2. git checkout -branchname for new component-
  3. git add and git commit -m "..."
  4. --DO NOT MERGE INTO MASTER BRANCH FIRST--
    • To be done one github instead
  5. git push origin -branchname for new component-
  6. On GitHub, compare and pull request to review changes on that branch
  7. We can add comments within the code before committing to a merge
  8. Merge pull request which merges into the master branch(if there is no conflicts with the base branch)
